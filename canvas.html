Authentication with Flask and MongoDB

Icon Progress Bar/Navigation (built in browser, hidden in app)

To have real authentication, an application needs a trusted way to check who someone is, and that trust almost always rests on a database. In an authentication system the database holds the user records — here, just an email address and a password hash — and every security decision the app makes traces back to what is (and deliberately is not) stored in those records.

In the previous lesson, we connected Flask to MongoDB without authentication. That showed how a Flask application stores and retrieves records. In this lesson, MongoDB becomes part of a login system: Flask handles the application logic — receiving form data, checking routes, creating sessions, deciding what each user may see — while MongoDB stores the user records. This is much closer to how a real web application works than a simple users.json file.

Why the app "naps" — static vs dynamic hosting

The first thing the embed shows is not the app at all — it is a wake-up page. The Flask app is hosted on Render's free tier, which stops the app's process after about 15 minutes without visitors and restarts it on the next request, which takes 30–60 seconds. That exposes an important difference between two kinds of hosting:

  • Dynamic hosting (Render): a process runs your Python code to build each response. If the process is asleep, it cannot produce any HTML — not even a “please wait” message.
  • Static hosting (GitHub Pages): a plain file is served as-is. There is no process to sleep, so the page appears instantly, always.

So the waiting room lives on the static host. The wake-up page appears immediately, then repeatedly asks the Flask app “are you up yet?” — and the moment the app answers, it redirects the browser to it:

async function ping() {
  try {
    await fetch(APP_URL, { mode: "no-cors", cache: "no-store" });
    return true;   // the server answered — it is awake
  } catch (err) {
    return false;  // still waking up (or unreachable)
  }
}

One subtlety: the two sites are different origins, and the Flask app sets no CORS headers, so the wake-up page is not allowed to read the app's response. It doesn't need to — mode: "no-cors" lets the request go through, and merely getting an answer proves the app is awake.

How the authentication works

1. The server is the authority on password strength

The signup form shows a live strength meter in the browser, but browser checks are a convenience, not a defence — anyone can bypass a form with a hand-written request. The rule that actually matters runs on the server, on every registration:

result = evaluate_password(password)
if not result["acceptable"]:
    flash(
        "Password is not strong enough. It still needs: "
        + ", ".join(result["unmet"]) + ".",
        "error",
    )
    return redirect(url_for("new_user"))

The rules themselves (8+ characters, lowercase, uppercase, a number, a symbol) are defined once, as data, and both the server check and the browser meter are driven from that single definition — so they can never disagree.

2. Only password hashes are stored — never passwords

users.insert_one(
    {
        "email": email,
        "password_hash": generate_password_hash(password),
    }
)

generate_password_hash applies a one-way, salted hash. One-way means the original password cannot be recovered from what is stored — even by us, even if the database leaks. Salted means two users with the same password still get different hashes, so an attacker cannot spot reused passwords or use precomputed tables. At login, check_password_hash re-hashes what the user typed and compares.

3. Logging in creates a signed session

user = users.find_one({"email": email})

if user is None or not check_password_hash(user["password_hash"], password):
    flash("Invalid email or password.", "error")
    return redirect(url_for("login"))

session["user_email"] = email

Two details here are deliberate. First, the error message is identical whether the email is unknown or the password is wrong — a login form that says “no such user” lets anyone probe which emails hold accounts. Second, session is stored in a cookie that Flask cryptographically signs with the app's SECRET_KEY: users can read their own cookie, but any tampering (say, editing the email inside it) breaks the signature and the session is rejected. This is why the secret key must be truly secret in production.

4. Routes that need a user are wrapped in a decorator

def login_required(view):
    @wraps(view)
    def wrapped(*args, **kwargs):
        if "user_email" not in session:
            flash("Please log in to continue.", "error")
            return redirect(url_for("login"))
        return view(*args, **kwargs)

    return wrapped

Any route decorated with @login_required bounces anonymous visitors to the login page before a single line of the route runs. Protection lives in one place instead of being copy-pasted into every view.

5. Users see their own data — and nothing else

@app.route("/users")
@login_required
def account():
    email = session["user_email"]
    current_user = users.find_one({"email": email})   # only YOUR record
    return render_template("account.html", current_user=current_user)

An earlier version of this page listed every user's email to any signed-in account — which meant one account could harvest the whole membership list. The fix queries only the signed-in user's own document. Authentication says who you are; deciding what you may see is a separate obligation, and the safe default is: your own data, nothing more.

Key takeaways

  • Store salted password hashes, never passwords — hashing is one-way on purpose.
  • Client-side validation is UX; the server-side check is the security boundary.
  • A session is a signed cookie: readable by the user, forgeable by no one — as long as the secret key is secret.
  • Error messages leak information; keep “wrong email” and “wrong password” indistinguishable.
  • Being logged in is not permission to see everything — default to showing users only their own records.
  • Dynamic apps can sleep; static files cannot. Put the “please wait” where it can always be served.
